For the **app-idea** profile, the canonical alternative solution shapes you should consider (not exhaustive — these are the seven that most often produce the strongest reframings):

1. **App** — the user's default frame: a downloadable mobile or web application. Surface it explicitly as a frame so we can argue against it.
2. **Service** — done-for-you human (or human-in-the-loop) operation. The user sends an input, a real person does the work, the user gets the output. Often dominates apps for low-frequency, high-pain jobs.
3. **Community** — Discord/Slack/forum where the value comes from peer interaction, not software features. Frequently dominates app frames for problems that are really loneliness or expertise-gap problems.
4. **Hardware** — a physical object (with minimal software) is the right answer. Surprisingly often the case for "ambient" or "trigger-based" jobs.
5. **Content product** — a newsletter, podcast, course, or playbook. The job is "I want to learn / be told / be reminded," and software is overkill.
6. **Marketplace** — connecting two sides who already exist (supply + demand) rather than building a tool one side uses alone.
7. **Managed service / agency** — the team builds the value as a service for a small number of high-paying clients before (or instead of) productizing.

When generating `alternative_shapes`, pick at least **two** of the seven that genuinely fit the user's JTBD. Reject the ones that don't fit — don't list all seven for completeness. Each chosen alternative must come with the mechanism, the audience shift, and what it's worse at, per the role spec.

Heretical takes for app-ideas tend to cluster around: "this should not be software," "this should be a feature in $existing-tool," "the user does not actually want this — they want to talk about wanting this," "the right MVP is a Google Sheet / a Calendly link / a Stripe Payment Link." If any of these apply, name them.
